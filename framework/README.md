# Framework Folder README

This folder contains the reusable framework code that runs inside tests for this project.
The framework provides step-based logging, assertion reporting, lifecycle management, and logger setup.

## Purpose

- Provide a shared test framework layer for all test packages.
- Capture test phase state for `setup`, `call`, and `teardown`.
- Log framework steps and assertion results in a structured way.
- Support consistent failure reporting across tests.

## Files and responsibilities

### `__init__.py`
- Marks `framework/` as a Python package.
- Allows import paths such as `from framework.step import step`.

### `assertion.py`
- Implements `assert_step(condition, message)`.
- If the condition is true, it logs a passing assertion step.
- If the condition is false, it logs a failed assertion step and raises a framework `AssertionError`.
- This ensures assertion failures are visible as step output in both logs and reports.

### `base_test.py`
- Provides `BaseTest`, a base class for test classes.
- Uses pytest fixtures to manage test lifecycle phases.
- The `lifecycle` fixture runs `setup()` and then the test method.
- The `_ensure_teardown` fixture ensures `teardown()` always runs, even if the test or setup failed.
- This ensures teardown steps are captured reliably.

### `logger.py`
- Configures the framework logger.
- Uses a custom formatter to include phase labels and timestamps.
- Writes framework step output into `output/test.log` by default.

### `step.py`
- Provides the public `step(message)` API for tests.
- Prints and logs step messages with current phase context.
- Records step events in memory for later report generation.
- Provides `clear_steps()` and `get_steps()` helpers.

### `config.py`
- Provides a central place for shared configuration values.
- Can be extended for environment flags, timeouts, or other common settings.

### `utils.py`
- Contains reusable helper functions for test logic.
- Typically used for validations, system checks, or environment utilities.

## How the framework works

1. A test inherits from `BaseTest`.
2. `BaseTest` sets the current phase and calls `setup()`, the test body, and `teardown()`.
3. `step(message)` records each action as a step and writes log output.
4. `assert_step()` logs failures before raising exceptions.
5. `tests/monitor_tests/run_tests.py` and `tests/conftest.py` collect step data and write final artifacts.

## Output behavior

- `output/test.log`
  - Contains timestamped step and phase output.
  - Includes an overall summary when test execution completes.

- `output/report.xml`
  - Contains a custom readable failure report with structured test step data.

- `output/pytest_report.xml`
  - Contains standard JUnit XML output generated by pytest.
